## 统一文档结构

每个模块的 `README.md` 应按以下顺序组织；暂时没有内容的章节可以标记“待设计”，但不要省略边界性信息：

1. **模块目标**
   - 说明模块解决的问题、使用者和成功标准。
2. **范围与非目标**
   - 明确本模块负责什么、不负责什么。
3. **职责与边界**
   - 说明与 `packages/core`、上层应用和相邻模块的职责分工。
4. **依赖关系**
   - 列出上游输入、下游消费者和允许依赖的包。
5. **公共接口**
   - 记录主要类型、函数、服务接口、命令或 HTTP API。
6. **数据模型与配置**
   - 记录关键实体、状态、持久化结构和配置格式。
7. **核心流程**
   - 描述正常流程、状态变化、数据流和必要的时序关系。
8. **异常与降级**
   - 描述错误传播、重试、幂等、超时、回滚和降级策略。
9. **安全与隐私**
   - 描述权限、脱敏、密钥处理、外部访问和审计要求。
10. **代码目录映射**
    - 映射真实业务目录、主要入口文件和关键实现位置。
11. **测试设计**
    - 列出单元测试、集成测试、端到端测试和关键验收场景。
12. **风险与待办**
    - 记录尚未解决的问题、兼容性风险和后续扩展点。

## 维护规则

- 模块文档必须与[总体架构](../02-架构/01-总体架构.md)中的模块边界一致。
- 业务目录发生新增、移动、重命名或删除时，同时更新[实现计划与代码结构](../02-架构/02-实现计划与代码结构.md)。
- 工具特定逻辑必须留在集成适配器中，不能进入 `packages/core`。
- 修改第三方配置的模块必须覆盖计划、备份、应用、验证和回滚。
- 涉及代理流量时必须保持流式响应兼容，不能缓冲面向用户的响应链路。
- 文档说明使用中文；代码标识、命令、协议名、产品名和文件路径保持原始形式。
